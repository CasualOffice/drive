# 12 — Content search surface

Companion to `02-surface.md` §"Top bar" and `docs/ARCHITECTURE.md` §"Content indexing". Doc-Hub's search reads *inside* documents, not just filenames. The question it answers is **"which document mentions X"** — you search for a phrase and get the documents that contain it, with the matching passage highlighted. Backed by the Rust `core` extraction engine + a Tantivy full-text index; an optional AI layer adds semantic search on top.

## Pattern reference

**Notion / GitHub code search / Dropbox** — typing in the search field flips the main pane from "this folder" to "matches across everything", and a full-text engine returns snippets with the hit highlighted. We reuse the existing document-list grid so context menu, multi-select, and the editor handoff all keep working. Search is a state flip inside `<Documents />`, not a separate route.

## What is indexed

On every committed version, the `dochub-index` background worker calls `core` to extract text from the document and writes it to Tantivy (per `ARCHITECTURE.md`):

- `docx`, `pptx` — body text, headings, notes.
- `xlsx`, `xlsm`, `csv` — cell values (opaque `xlsm` macros are not executed; only stored text is indexed).
- `pdf` — the text layer (OCR fallback is a later addition).
- `md`, `txt`, `json`, `yaml` — raw text.

A `files.index_state` column (`pending | ready | unsupported | failed`) drives the worker and the UI's "still indexing" affordance. Indexing runs on plaintext the server already decrypts to serve the editor — the deliberate server-trusted trade that makes content search possible (it is not zero-knowledge E2E). A tombstoned document is removed from the index.

## Behaviour

- Search input lives in the top bar.
- Empty query → the current folder, unchanged.
- Query ≥ 2 chars → `<Documents />` calls `GET /api/search?q=…&limit=50` and renders content matches instead of the folder listing.
- Header flips from the project/folder name to **`Search results`** with a count chip and the matched query echoed.
- Each result row shows the document, its project/folder path, and the **best-matching snippet** with the query terms wrapped in `<mark>` (theme token `--accent-muted` colours the highlight). Long documents show up to 3 snippets; a "+N more matches" affordance expands the rest inline.
- Click a document result → opens it in the editor, scrolled to the first match where the SDK supports a jump-to-offset. Click a folder result → navigate into it (clears search).
- Documents still `pending`/`failed` in the index are matched by name only and carry a small muted "indexing…" / "not indexed" tag so the absence of a content hit is never mistaken for "no match".
- Debounce: 200 ms after the last keystroke; in-flight request cancelled on new input via an AbortController.

## Backend contract

### `GET /api/search?q=<query>&limit=<n>` (authed, project-scoped)

```json
{
  "documents": [
    {
      "id": "doc_01H…",
      "name": "Q3 roadmap.docx",
      "path": "Engineering / Planning",
      "kind": "doc",
      "version_seq": 7,
      "index_state": "ready",
      "matches": [
        { "field": "content", "snippet": "…migrate the <mark>billing</mark> service by…", "score": 8.4 },
        { "field": "content", "snippet": "…<mark>billing</mark> owners: Alex, Sam…", "score": 6.1 }
      ]
    }
  ],
  "folders": [ { "id": "…", "name": "Billing", "matches": [ { "field": "name", "snippet": "<mark>Billing</mark>" } ] } ]
}
```

- `q` trimmed server-side; empty queries return empty arrays without touching Tantivy.
- `limit` clamped `[1, 200]`, default 50.
- **Match strategy:** Tantivy full-text over extracted `content` (highest weight) unioned with a `name` match, ranked by BM25 score. Phrase queries (`"exact phrase"`) and per-term stemming come from Tantivy's query parser.
- **Scope:** results are restricted to projects/workspaces the caller is a member of; membership is enforced per-hit (`WorkspaceMemberRepo::role_of`). Cross-project isolation is property-tested.
- Tombstoned documents are excluded (they're removed from the index on tombstone).
- Snippets are generated by Tantivy's `SnippetGenerator`; offsets are pre-wrapped in `<mark>` server-side so the client only renders sanitized fragments.

## State checklist

| | Required | Notes |
|---|---|---|
| Default (query 0–1 chars) | yes | renders the current folder |
| Loading | yes | grid skeleton; no flicker between keystrokes (debounce + AbortController) |
| Results | yes | grid/list of documents with snippets, "Search results · N" header |
| Result still indexing | yes | name-only match + "indexing…" tag |
| Empty | yes | EmptyState, `No document mentions "<query>".` |
| Error | yes | inline ErrorState ("Couldn't reach the search index.") |
| Index unavailable (worker down) | yes | falls back to name-only match + a one-line "content search is offline" banner |

## Refinement — filters, ranking, suggestions

The floor is "type a phrase, get the documents that contain it, snippet-highlighted." The next pass makes it a tool a compliance or engineering team leans on daily: filter chips over the same result grid, and keyboard-first navigation.

### Filter surface

A horizontal row of chips between the `Search results · N` header and the result grid; chips appear only after the user has typed (or entered search mode via the palette):

| Chip | Control | Maps to |
|---|---|---|
| **Type** | multi-select: Document / Spreadsheet / Presentation / PDF / Markdown / Text / Data (json/yaml/csv) | canonical `kind` buckets (documents only — no image/video/audio/archive buckets) |
| **Project** | multi-select; defaults to the current project | `workspace_id`; omitted when only one is visible |
| **Author** | autocomplete from project members | the version author (`file_versions.author_id`) |
| **Modified** | date-range with relative shortcuts (Today / Last 7 / Last 30 / Custom) | head version `created_at` |
| **Has share link** | toggle | share-link state |
| **Under legal hold** | toggle | compliance filter — find held records fast |
| **In trash** | toggle (default off) | tombstoned documents |

Active filters render the chip filled/accented with the value summarised inline (`Type: PDF, Spreadsheet`, `Modified: Last 7 days`). `Clear all` appears at the row's right when ≥ 1 filter is active. URL state carries query + filters so a result view is a shareable link (`/search?q=billing&type=pdf`).

### Ranking + snippets

- Content hits rank by Tantivy BM25 `_score`; a name hit is boosted so an exact filename match sits near the top.
- Each result shows up to 3 snippets; `<mark>` offsets come pre-computed from the backend.
- Sort options (default **Relevance**): Modified ↓/↑, Name A→Z/Z→A, Versions ↓ (most-edited first). Folder-grouping (folders first) applies only to Name sort. Sort persists in `localStorage` (`dochub-search-sort-v1`) and resets to Relevance when the query is cleared.

### Suggested-when-empty

Search field focused, query empty → a curated grid instead of the empty state:

1. **Recently opened** — last 8 documents the user opened in the editor (audit-log lookup, 30-day window).
2. **Recently edited by others** — documents in the project committed by other members in the last 24h (skipped for the personal locker).
3. **Needs attention** — any document with a failed index or a chain-verification flag, so integrity problems surface where people look.

### No-results recovery

```
                     No document mentions "kickoff"

       Try:
         ⤴ Search in trash too          [toggle In-trash filter]
         ⤴ Drop the Author filter        [if Author set]
         ⤴ Widen Modified to All         [if Modified set]
         ⤴ Search across all projects    [if scope = current]

       Did you mean: "kickoffs", "kick off"?   [Tantivy fuzzy suggestions]
```

Each line is a button that relaxes the named filter and re-runs in place. Suggestions come from a Tantivy fuzzy/`Levenshtein` term lookup over the index vocabulary.

### Pagination

Cursor pagination with infinite scroll (desktop) + "Load more" (mobile / keyboard). The server returns an opaque, HMAC-signed `next_cursor` encoding `(sort_field, last_value, last_id)` so it can't be tampered to leak across projects. Default 30/page, clamp `[10, 100]`. An `IntersectionObserver` sentinel ~2 viewports below the last result triggers the next page; concurrent fetches coalesce via the AbortController pattern. `— End of results —` divider when `next_cursor` is null.

### Keyboard model

- `Cmd-K` → palette overlay (filter chips render below its input).
- `/` → focus the top-bar search input.
- `Tab` from the input → first chip; `Shift-Tab` back.
- `Esc` from a popover → close; from a focused chip → clear that filter; from the input with no query → exit search mode.

### Accessibility

- Input is `role="combobox" aria-expanded` driven by the suggestion popover.
- Result count is announced via `aria-live="polite"` ("142 documents, 6 folders").
- Filter chips are `role="button" aria-pressed`; popovers are `role="dialog"` trapping focus.
- Fully keyboard-operable; every icon-only control has a label + tooltip.
- Respects `prefers-reduced-motion`: no scroll-into-view or slide-in animation.

### Performance budget

| Operation | Target | Backstop |
|---|---|---|
| Keystroke → first paint (after debounce) | < 200 ms p95 | 300 ms hard timeout → stale results + spinner |
| Pagination round-trip | < 200 ms p95 | 500 ms → keep skeleton row |
| Filter-chip change → re-search | < 200 ms p95 | same |
| AbortController cancels in-flight on every keystroke | always | never render a stale response |

Cumulative layout shift across a session: 0 — skeletons match post-load row geometry.

## Optional AI layer — semantic search

When `dochub-ai` is enabled, a semantic layer sits **beside** Tantivy, never replacing it for compliance-critical retrieval (`ARCHITECTURE.md` §"AI layer"):

- **Semantic search** — embeddings + rerank surface documents that are *about* the query even when they don't contain the literal terms ("severance terms" finds a contract that says "termination compensation"). Results merge with the Tantivy hits, tagged so the user can see which matched by content and which by meaning.
- **Cross-document Q&A** — a natural-language question ("which contracts auto-renew?") returns an answer with **citations to the source documents and passages**; every cited claim links to the exact version + snippet. The answer is read-only and never mutates a document or the history.
- **Query rewriting** — "PDFs Alex changed last week" → structured filters (`type: pdf, author: alex, modified_after: 7d`).
- **Provider:** pluggable; default Claude via the Anthropic API (Haiku for extraction/classification, Sonnet/Opus for reasoning/Q&A), local-model adapter for air-gapped installs. Every AI query is audited.

The AI toggle is a mode switch on the search surface (`Full-text` ⇄ `Semantic`), defaulting to full-text so the deterministic, verifiable path is always the baseline. Absent `dochub-ai`, the switch is hidden and Tantivy full-text is the whole surface.

## Out of scope

- OCR of image-only PDFs — later index-worker enhancement.
- Boolean query language beyond phrase quoting (`-not`, `OR`, `author:` bare words) — the autocomplete groundwork is laid; parsing isn't.
- Saved / shared searches (server-side persistence).
- Search-as-API for external integrations.
