## The big picture

```
                    ┌───────────────────────────────┐
                    │           Browser             │
                    └────────────┬──────────────────┘
                                 │
                ┌────────────────┼─────────────────────┐
                │                │                     │
        hub.example.com        │     usercontent-dochub.example.com
        ┌─────────────────┐      │     ┌──────────────────────────┐
        │  SPA · JSON API │      │     │  /raw/{token} (bytes)    │
        │  Editor streams │      │     │  CSP: sandbox; none      │
        │  Session cookie │      │     │  Content-Disposition:    │
        │  Strict CSP     │      │     │    attachment            │
        └────────┬────────┘      │     └─────────────┬────────────┘
                 │               │                   │
                 └───────────────┼───────────────────┘
                                 │
                       ┌─────────▼─────────┐
                       │  hub binary     │
                       │  (Rust + Axum)    │
                       └─────────┬─────────┘
                                 │
       ┌──────────┬──────────────┼──────────────┬──────────┐
       │          │              │              │          │
   SQL DB    Encrypted        Version +      Index /    Embedded
 (SQLite/    storage          hash-chain      AI        editors
  Postgres)  facade           engine       (Tantivy)  (Sheet/Docs/PDF)
```

A single Rust binary. Two HTTP origins served by the same process — the request `Host` header selects the surface. A mandatory encryption layer sits in front of every storage backend. Every save appends a hash-chained version; nothing is ever overwritten. The server holds the keys — a deliberate trade that makes content search and AI possible.

## Encrypted storage facade

Handler code talks to `Arc<Storage>`. It never reaches into the OpenDAL operator directly. The facade wraps the operator with a mandatory encryption layer:

```
write(key, plaintext)  →  seal(workspace_dek, plaintext)  →  operator.write(key, ciphertext)
read(key)              →  operator.read(key) → open(workspace_dek, ciphertext) → plaintext
```

- **Envelope encryption.** Each workspace has a Data Encryption Key (DEK). The DEK is wrapped by a master Key Encryption Key (KEK) supplied via `DOCHUB_MASTER_KEY` or an external KMS; only wrapped DEKs are persisted. Primitive: AES-256-GCM (`ring`/`aws-lc-rs`), a random 96-bit nonce per blob, stored as `nonce ‖ ciphertext ‖ tag`.
- **No plaintext at rest.** No code path writes plaintext document bytes to a backend. Enforced by construction — handlers can't reach the raw operator — and by test, where a spy backend asserts ciphertext.
- **Key rotation** re-wraps DEKs under a new KEK without rewriting document blobs.
- **Backends:** `fs`, `memory`, `s3`, `minio`, `r2`, `b2` via OpenDAL. Storage keys are opaque ULIDs, never derived from user input. BYO-bucket credentials are sealed with the same envelope scheme.

Encryption defends a stolen disk or database dump — not a fully compromised, running server. That's the boundary, stated plainly. This is **not** zero-knowledge E2E.

## Immutable version + hash-chain engine

The heart of the registry.

```
file_versions(file_id, seq, storage_key, size, content_hash, prev_hash, author_id, reason, created_at)
```

- On every committed save, a **new** version row is appended: `content_hash = SHA-256(ciphertext)`, `prev_hash =` the previous version's `content_hash`. `seq` increments; the head is the file's current version.
- **Write-once.** Version blobs are content-addressed and never overwritten. "Delete" writes a tombstone and obeys retention + legal hold; bytes under hold are never removed.
- **Verification.** `verify_chain(file_id)` recomputes hashes and links end-to-end. A mismatch is a tamper alarm — surfaced to admins and audited, never silently repaired.
- **Restore** version *k* appends a new version *N+1* whose bytes equal *k*. The old chain is preserved; restore is additive.
- **Audit chain.** The append-only `audit_log` is hash-chained the same way; committed rows are never updated or deleted. Optional: periodic Ed25519-signed anchoring of chain heads for third-party-verifiable provenance.

## Two-origin model

The app and the document bytes live on different origins. This is non-negotiable.

- **`hub.example.com`** — the SPA, the JSON API, the embedded-editor byte streams. Cookies live here. A strict CSP locks the page down.
- **`usercontent-dochub.example.com`** — the only place raw bytes are ever served, via `/raw/{token}` for share links and isolated content. `Content-Security-Policy: sandbox; default-src 'none'`. No cookies. Every non-previewable type gets `Content-Disposition: attachment` forced.

The hub **refuses to start** in production if the two origins are configured the same. Neither CSP is weakened; `/raw/{token}` never moves to the app origin.

## Four tokens, four jobs

Don't reuse one for another's job.

| Token | Where it lives | Purpose | TTL |
|---|---|---|---|
| **Session cookie** | `__Host-dh_sid` browser cookie | The signed-in user | configurable (default 7d) |
| **Editor access token** | Header on every editor byte-stream request | Per-launch, per-document edit grant, HMAC over `(user_id, file_id, perms, exp, jti)` | short |
| **Share-link token** | Path segment `/s/<token>` | Public, optionally-password-gated share | configurable |
| **Signed-URL token** | `?token=` on `/raw/{token}` | Per-download HMAC over `(key, exp, method)` for the fs/mem backends | short |

All are validated server-side with constant-time compares. The editor access token's document id in the URL must match the id in the claim, every call.

## Embedded editing

The primary editing path is **embedded**, not WOPI:

```
SPA opens a document → app origin mints an editor access token → embedded editor (Sheet/Docs/PDF)
   → server decrypts bytes in memory, streams them over the authenticated app origin
   → user edits (optionally co-edits via the collab server)
   → save → encrypt → append hash-chained version → audit → enqueue reindex
```

One `<Editor>` component hosts each format via its sibling SDK. Team documents co-edit through the `collab` server (Yjs / Hocuspocus), which relays opaque document bytes and never parses them. WOPI remains available as **optional interop** for external Office clients, but is not the default path.

## Content indexing

`dochub-index` runs a lazy background worker: on a new version, it calls the `core` engine to extract text (`docx`/`xlsx`/`pdf`/`md`/`txt`/`csv`/`json`/`yaml`), normalizes it, and writes to a Tantivy full-text index. A `files.index_state` column (`pending | ready | unsupported | failed`) drives the worker. Search unions Tantivy content hits with SQL metadata and returns snippets + highlights — it reads *inside* documents, not just filenames. Tombstoning a document removes it from the index.

## AI layer (optional)

`dochub-ai` sits beside search, never replacing it for compliance-critical retrieval:

- **Semantic search** — embeddings + rerank alongside the Tantivy exact/full-text results.
- **Summaries, entity + PII detection, cross-document Q&A** — all read-only suggestions; a human approves any action. AI never mutates documents or history.
- **Provider** — pluggable; default Claude via the Anthropic API (Haiku for extraction/classification, Sonnet/Opus for reasoning), with a local-model adapter for air-gapped installs. Every AI action is audited.

## Boot invariants (fail-fast)

The binary refuses to start unless: a master key or KMS is configured; the two origins differ in production; the DB migrates cleanly; the storage backend is reachable. These are asserted in `dochub-bin` and covered by tests.
